# Operations: Backup and Restore

Everything you need to back up a Talos Kubernetes cluster and recover from failures — from single-node loss to full cluster rebuild.

---

## What to Back Up

| Item | Contains | How to Back Up | Frequency |
|------|----------|---------------|-----------|
| **etcd** | All Kubernetes state (pods, services, secrets, configmaps, CRDs) | `talosctl etcd snapshot` | Hourly for prod, daily for dev |
| **Machine configs** | Per-node Talos configuration (network, disk, extensions) | `talosctl get mc -o yaml` | After every config change |
| **secrets.yaml** | Cluster CA keys, bootstrap token, etcd encryption key | Copy the file generated by `talosctl gen secrets` | Once (store securely, never lose) |
| **Kubernetes resources** | Application manifests, PVCs, namespaces, RBAC | Velero or manual `kubectl get -o yaml` | Daily or per-policy |
| **Persistent volumes** | Application data on CSI volumes | Velero + CSI snapshots, or storage-level backup | Per application RPO |

**The most critical item is `secrets.yaml`.** Without it, you cannot regenerate machine configs that are compatible with your cluster's PKI. If you lose `secrets.yaml` and all nodes simultaneously, the cluster is unrecoverable.

## etcd Backup

### Manual Snapshot

```bash
# Take an etcd snapshot — target any control-plane node
talosctl etcd snapshot db.snapshot --nodes 192.168.1.10

# Verify the snapshot file
ls -la db.snapshot
# Should be non-zero size, typically 2-50 MB for most clusters
```

The snapshot contains the entire etcd keyspace — every Kubernetes object, every secret, every CRD instance.

### Automated Backup Script

Save as `scripts/etcd-backup.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail

# Configuration
CP_NODE="192.168.1.10"
BACKUP_DIR="/backups/etcd"
RETAIN_DAYS=30

# Create backup directory
mkdir -p "$BACKUP_DIR"

# Generate timestamped filename
TIMESTAMP=$(date -u +%Y%m%d-%H%M%S)
SNAPSHOT_FILE="${BACKUP_DIR}/etcd-snapshot-${TIMESTAMP}.db"

# Take the snapshot
echo "[$(date -u)] Taking etcd snapshot..."
talosctl etcd snapshot "$SNAPSHOT_FILE" --nodes "$CP_NODE"

# Verify snapshot is non-empty
SNAPSHOT_SIZE=$(stat -f%z "$SNAPSHOT_FILE" 2>/dev/null || stat -c%s "$SNAPSHOT_FILE" 2>/dev/null)
if [ "$SNAPSHOT_SIZE" -lt 1000 ]; then
    echo "ERROR: Snapshot file is suspiciously small (${SNAPSHOT_SIZE} bytes)"
    exit 1
fi

echo "[$(date -u)] Snapshot saved: $SNAPSHOT_FILE ($SNAPSHOT_SIZE bytes)"

# Optional: copy to remote storage
# aws s3 cp "$SNAPSHOT_FILE" "s3://my-backups/etcd/"
# rclone copy "$SNAPSHOT_FILE" remote:backups/etcd/

# Clean up old snapshots
find "$BACKUP_DIR" -name "etcd-snapshot-*.db" -mtime +"$RETAIN_DAYS" -delete
echo "[$(date -u)] Cleaned snapshots older than $RETAIN_DAYS days"
```

```bash
chmod +x scripts/etcd-backup.sh

# Run hourly via cron (on your workstation or a bastion host):
# 0 * * * * /path/to/scripts/etcd-backup.sh >> /var/log/etcd-backup.log 2>&1
```

## Machine Config Backup

### Export Current Running Config

```bash
# Export machine config from each node
talosctl get mc -o yaml --nodes 192.168.1.10 > backups/mc-cp-1.yaml
talosctl get mc -o yaml --nodes 192.168.1.11 > backups/mc-cp-2.yaml
talosctl get mc -o yaml --nodes 192.168.1.12 > backups/mc-cp-3.yaml
talosctl get mc -o yaml --nodes 192.168.1.20 > backups/mc-worker-1.yaml
talosctl get mc -o yaml --nodes 192.168.1.21 > backups/mc-worker-2.yaml
```

**Note:** The exported config contains the full running configuration including all applied patches. Store these alongside your original `controlplane.yaml`, `worker.yaml`, and patch files.

### What to Keep in Version Control

```
cluster-config/
  secrets.yaml          # ENCRYPTED (use age, sops, or sealed-secrets — never plaintext in git)
  controlplane.yaml     # Base control-plane config
  worker.yaml           # Base worker config
  patches/
    cp-1.yaml           # Per-node patches
    cp-2.yaml
    cp-3.yaml
    worker-1.yaml
    worker-2.yaml
  talosconfig           # Client config (contains client certs — encrypt or exclude)
```

## Emergency etcd Snapshot

If the etcd cluster is unhealthy and `talosctl etcd snapshot` fails, you can copy the raw database file directly:

```bash
# Copy the raw etcd database from a CP node
talosctl cp /var/lib/etcd/member/snap/db ./emergency-etcd.db --nodes 192.168.1.10

# This is a raw BoltDB file — it may not be fully consistent if etcd was mid-write
# Prefer talosctl etcd snapshot when possible — it creates a consistent snapshot
```

**When to use this:**
- `talosctl etcd snapshot` fails because etcd refuses connections
- etcd is in a crash loop but the data directory is intact
- You need a best-effort backup before a risky recovery operation

## Disaster Recovery Scenarios

### Scenario 1: Single Node Failure (Quorum Intact)

**Situation:** One CP or worker node is lost, but the etcd cluster still has quorum (2 of 3 CP nodes running).

**Recovery — replace the failed node:**

```bash
# 1. Remove the failed node from etcd (if it was a CP node)
talosctl etcd members --nodes 192.168.1.11
# Identify the failed member ID
talosctl etcd remove-member <member-id> --nodes 192.168.1.11

# 2. Delete the Kubernetes node object
kubectl delete node <failed-node-name>

# 3. If reusing the same machine, wipe and re-apply config
talosctl reset --nodes 192.168.1.10 --graceful=false --reboot

# Wait for the node to come back in maintenance mode, then re-apply config
talosctl apply-config --insecure --nodes 192.168.1.10 --file cp-1.yaml

# 4. If replacing with a new machine, boot from Talos media and apply config
talosctl apply-config --insecure --nodes <new-ip> --file cp-replacement.yaml

# 5. Verify
talosctl etcd members --nodes 192.168.1.11
talosctl health --nodes 192.168.1.11
kubectl get nodes
```

**Recovery — reset EPHEMERAL partition (data corruption, not hardware failure):**

```bash
# Reset only the EPHEMERAL partition (etcd data, kubelet state)
# The node will reboot and re-join the etcd cluster, syncing from healthy members
talosctl reset --nodes 192.168.1.10 \
  --graceful \
  --system-labels-to-wipe EPHEMERAL

# The node re-applies its config, kubelet and etcd start fresh
# etcd automatically re-joins and replicates from the other members
# Wait for it to complete
talosctl etcd members --nodes 192.168.1.11
```

### Scenario 2: Quorum Loss (Majority of CP Nodes Down)

**Situation:** 2 of 3 CP nodes are down. etcd cannot elect a leader. The Kubernetes API server is unavailable.

**Recovery from etcd snapshot:**

```bash
# 1. Identify the one remaining CP node (or pick any CP node if all are up but etcd is broken)
# 2. Ensure you have a recent etcd snapshot file (db.snapshot)

# 3. Reset the surviving CP node to clear its etcd state
talosctl reset --nodes 192.168.1.10 \
  --graceful=false \
  --system-labels-to-wipe EPHEMERAL

# 4. Wait for the node to come back, then re-apply config
talosctl apply-config --insecure --nodes 192.168.1.10 --file cp-1.yaml

# 5. Bootstrap the node, recovering from the snapshot
talosctl bootstrap --nodes 192.168.1.10 \
  --recover-from=./db.snapshot

# 6. Wait for Kubernetes to come up
# The API server will start with the state from the snapshot
talosctl health --nodes 192.168.1.10 --wait-timeout 300s

# 7. Bring the other CP nodes back
# Reset and re-apply config — they will join the recovered etcd cluster
talosctl reset --nodes 192.168.1.11 --graceful=false --system-labels-to-wipe EPHEMERAL
talosctl apply-config --insecure --nodes 192.168.1.11 --file cp-2.yaml
# Repeat for cp-3

# 8. Verify full cluster recovery
talosctl etcd members --nodes 192.168.1.10
talosctl etcd status --nodes 192.168.1.10
kubectl get nodes
kubectl get pods --all-namespaces
```

---

### Scenario 3: Full Cluster Rebuild

**Situation:** All nodes are gone. You are rebuilding from scratch using backed-up artifacts.

**Required artifacts:**
- `secrets.yaml` — without this, recovery is impossible (new PKI will not match)
- `db.snapshot` — etcd snapshot containing all Kubernetes state
- Machine configs or the ability to regenerate them from `secrets.yaml`

**Procedure:**

```bash
# 1. Regenerate machine configs using the SAME secrets.yaml
talosctl gen config my-cluster https://192.168.1.100:6443 \
  --with-secrets secrets.yaml

# 2. Apply per-node patches as before
talosctl machineconfig patch controlplane.yaml --patch @patches/cp-1.yaml --output cp-1.yaml
talosctl machineconfig patch controlplane.yaml --patch @patches/cp-2.yaml --output cp-2.yaml
talosctl machineconfig patch controlplane.yaml --patch @patches/cp-3.yaml --output cp-3.yaml

# 3. Boot nodes from Talos media and apply configs
talosctl apply-config --insecure --nodes 192.168.1.10 --file cp-1.yaml
talosctl apply-config --insecure --nodes 192.168.1.11 --file cp-2.yaml
talosctl apply-config --insecure --nodes 192.168.1.12 --file cp-3.yaml

# 4. Bootstrap the first CP node with the etcd snapshot
talosctl bootstrap --nodes 192.168.1.10 \
  --recover-from=./db.snapshot

# 5. Wait for the cluster to come up
talosctl health --nodes 192.168.1.10 --wait-timeout 600s

# 6. Apply worker configs
talosctl apply-config --insecure --nodes 192.168.1.20 --file worker-1.yaml
talosctl apply-config --insecure --nodes 192.168.1.21 --file worker-2.yaml

# 7. Verify complete recovery
talosctl etcd members --nodes 192.168.1.10
kubectl get nodes
kubectl get pods --all-namespaces

# 8. Restore persistent volume data using Velero (if applicable)
velero restore create --from-backup <backup-name>
```

## Velero Integration

Velero backs up Kubernetes resources and persistent volumes to object storage. It complements etcd snapshots by providing application-level backup and restore with namespace granularity.

### Install Velero

```bash
# Add the Velero Helm repo
helm repo add vmware-tanzu https://vmware-tanzu.github.io/helm-charts
helm repo update

# Install with S3-compatible backend
helm install velero vmware-tanzu/velero \
  --namespace velero \
  --create-namespace \
  --version 7.2.1 \
  --set configuration.backupStorageLocation[0].name=default \
  --set configuration.backupStorageLocation[0].provider=aws \
  --set configuration.backupStorageLocation[0].bucket=my-velero-backups \
  --set configuration.backupStorageLocation[0].config.region=us-east-1 \
  --set configuration.backupStorageLocation[0].config.s3ForcePathStyle=true \
  --set configuration.backupStorageLocation[0].config.s3Url=https://s3.example.com \
  --set configuration.volumeSnapshotLocation[0].name=default \
  --set configuration.volumeSnapshotLocation[0].provider=aws \
  --set configuration.volumeSnapshotLocation[0].config.region=us-east-1 \
  --set credentials.secretContents.cloud="$(cat velero-credentials)" \
  --set initContainers[0].name=velero-plugin-for-aws \
  --set initContainers[0].image=velero/velero-plugin-for-aws:v1.10.1 \
  --set initContainers[0].volumeMounts[0].mountPath=/target \
  --set initContainers[0].volumeMounts[0].name=plugins
```

The credentials file format:

```ini
# velero-credentials
[default]
aws_access_key_id=AKIAIOSFODNN7EXAMPLE
aws_secret_access_key=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
```

### Backup Schedules

```bash
# Create a daily backup of the entire cluster
velero schedule create daily-full \
  --schedule="0 2 * * *" \
  --ttl 720h

# Create an hourly backup of critical namespaces
velero schedule create hourly-critical \
  --schedule="0 * * * *" \
  --include-namespaces production,databases \
  --ttl 168h

# Create a backup with volume snapshots
velero schedule create daily-with-volumes \
  --schedule="0 3 * * *" \
  --default-volumes-to-fs-backup=true \
  --ttl 720h
```

### Manual Backup

```bash
# Backup a specific namespace
velero backup create my-app-backup \
  --include-namespaces my-app \
  --default-volumes-to-fs-backup=true

# Check backup status
velero backup describe my-app-backup
velero backup logs my-app-backup
```

### Restore

```bash
# List available backups
velero backup get

# Restore from a backup
velero restore create --from-backup my-app-backup

# Restore specific namespaces from a full backup
velero restore create --from-backup daily-full-20250115020000 \
  --include-namespaces my-app

# Check restore status
velero restore describe <restore-name>
velero restore logs <restore-name>
```

### Verify Backups

```bash
# Check that scheduled backups are running
velero schedule get
velero backup get --sort-by .metadata.creationTimestamp

# Check backup storage location status
velero backup-location get
```

## Backup Strategy Summary

| Cluster Type | etcd Snapshots | Machine Configs | Velero | secrets.yaml |
|-------------|---------------|----------------|--------|-------------|
| **Production** | Hourly, 30-day retention, off-site | After every change, in version control | Daily full + hourly critical namespaces | Encrypted, multiple secure locations |
| **Staging** | Daily, 7-day retention | In version control | Weekly or none | Encrypted, one secure location |
| **Dev/Lab** | Before upgrades only | In version control | None | Local encrypted file |
