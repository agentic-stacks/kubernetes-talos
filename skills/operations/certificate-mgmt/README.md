# Operations: Certificate Management

Talos manages two independent certificate authorities (CAs). This skill covers their architecture, rotation procedures, and expiry monitoring.

---

## Certificate Architecture

Talos clusters use two separate PKI hierarchies:

### 1. Machine CA (Talos API)

- **Purpose:** Secures all Talos API (gRPC) communication — `talosctl` to nodes, node-to-node trust
- **Generated by:** `talosctl gen secrets` (stored in `secrets.yaml` as `certs.os`)
- **Used by:** `apid`, `trustd`, `talosctl` client certificates
- **Default lifetime:** 10 years from cluster creation
- **Port:** TCP 50000

The `talosconfig` file contains a client certificate signed by this CA. When you run any `talosctl` command, the client cert authenticates you to `apid` on the target node.

### 2. Cluster CA (Kubernetes API)

- **Purpose:** Secures all Kubernetes API communication — `kubectl` to API server, kubelet to API server, service account tokens
- **Generated by:** `talosctl gen secrets` (stored in `secrets.yaml` as `certs.k8s`)
- **Used by:** `kube-apiserver`, `kubelet`, `kube-controller-manager`, `kubectl` client certificates
- **Default lifetime:** 10 years from cluster creation
- **Port:** TCP 6443

The `kubeconfig` file contains a client certificate (or references a token) signed by this CA.

### Certificate Hierarchy Diagram

```
secrets.yaml
  |
  +-- certs.os (Machine CA)
  |     +-- apid server cert (per node)
  |     +-- talosctl client cert (in talosconfig)
  |     +-- trustd cert (per node)
  |
  +-- certs.k8s (Cluster CA)
  |     +-- kube-apiserver server cert
  |     +-- kubelet client cert (per node)
  |     +-- controller-manager client cert
  |     +-- scheduler client cert
  |     +-- kubectl client cert (in kubeconfig)
  |     +-- service account signing key
  |
  +-- certs.etcd (etcd CA)
        +-- etcd peer certs
        +-- etcd client certs (used by API server)
```

## When to Rotate Certificates

| Trigger | Action Required |
|---------|----------------|
| **Compromise** — CA private key or any signed cert is leaked | Immediate rotation of the affected CA |
| **Revocation** — need to revoke access for a specific client cert | Rotate the CA (Talos does not support individual cert revocation) |
| **Expiry approaching** — within 1 year of the 10-year expiry | Proactive rotation |
| **Compliance** — organizational policy requires shorter cert lifetimes | Scheduled rotation per policy |
| **Personnel change** — admin with `talosconfig` access leaves the organization | Rotate Machine CA to invalidate their client cert |

## Talos API CA Rotation

Rotates the Machine CA that secures `talosctl` communication. This does **not** affect Kubernetes API access.

### Dry Run First

```bash
# Preview what will happen — no changes are made
talosctl rotate-ca \
  --talos=true \
  --kubernetes=false \
  --dry-run \
  --nodes 192.168.1.10,192.168.1.11,192.168.1.12 \
  --output new-talosconfig
```

### Execute Rotation

```bash
# Rotate the Talos (Machine) CA
talosctl rotate-ca \
  --talos=true \
  --kubernetes=false \
  --nodes 192.168.1.10,192.168.1.11,192.168.1.12 \
  --output new-talosconfig
```

**What happens during rotation:**
1. A new Machine CA is generated
2. Each node transitions to accept certificates from both old and new CAs (dual-trust phase)
3. New server and client certificates are issued under the new CA
4. The old CA is removed from the trust bundle
5. A new `talosconfig` is written to `--output` containing the new client certificate

**Impact:** Non-disruptive. The dual-trust phase ensures zero downtime. Kubernetes API, workloads, and pod networking are completely unaffected.

### Post-Rotation Steps

```bash
# Replace your local talosconfig with the new one
cp new-talosconfig ~/.talos/config

# Verify connectivity
talosctl version --nodes 192.168.1.10
talosctl health --nodes 192.168.1.10

# Distribute the new talosconfig to all administrators
# The old talosconfig is now invalid

# Update secrets.yaml backup with the new CA
# (the new CA is embedded in the talosconfig and in the node machine configs)
```

## Kubernetes API CA Rotation

Rotates the Cluster CA that secures `kubectl` communication and all internal Kubernetes PKI.

### Dry Run First

```bash
talosctl rotate-ca \
  --talos=false \
  --kubernetes=true \
  --dry-run \
  --nodes 192.168.1.10,192.168.1.11,192.168.1.12
```

### Execute Rotation

```bash
talosctl rotate-ca \
  --talos=false \
  --kubernetes=true \
  --nodes 192.168.1.10,192.168.1.11,192.168.1.12
```

**What happens during rotation:**
1. A new Kubernetes CA is generated
2. API server, kubelet, and other components transition to dual-trust
3. New server and client certificates are issued under the new CA
4. The old CA is removed from the trust bundle
5. All kubelets re-register with new certificates

**Impact:** May cause **brief disruption** to Kubernetes API connectivity during the transition. Expect:
- Short periods where `kubectl` commands fail (seconds to ~1 minute)
- Pods using in-cluster service account tokens may need to refresh (automatic in most cases)
- Webhook and API aggregation layer endpoints may briefly fail validation

### Post-Rotation Steps

```bash
# Regenerate kubeconfig with the new CA
talosctl kubeconfig --nodes 192.168.1.10 --force

# Verify connectivity
kubectl get nodes
kubectl get pods -n kube-system

# Distribute the new kubeconfig to all users
# The old kubeconfig is now invalid

# Restart any webhook or aggregation-layer services that cache CA bundles
kubectl rollout restart deployment -n cert-manager cert-manager-webhook 2>/dev/null || true
```

## Manual Certificate Rotation (Advanced)

If `talosctl rotate-ca` is not available (older Talos versions) or you need finer control, use the 5-phase manual process.

**This is complex and error-prone. Use `talosctl rotate-ca` whenever possible.**

### Phase 1: Generate New CA

```bash
# Generate a new secrets bundle
talosctl gen secrets -o new-secrets.yaml

# Extract just the new CA certificates you need
# The new-secrets.yaml contains fresh CA keys for both Talos and Kubernetes
```

### Phase 2: Add New CA to Trust Bundle (Dual Trust)

```yaml
# patch-dual-trust.yaml
# Add the new CA as an accepted CA alongside the existing one
machine:
  acceptedCAs:
    - |
      -----BEGIN CERTIFICATE-----
      <new CA certificate PEM>
      -----END CERTIFICATE-----
```

```bash
# Apply to all nodes
talosctl patch machineconfig --nodes 192.168.1.10 --patch @patch-dual-trust.yaml
talosctl patch machineconfig --nodes 192.168.1.11 --patch @patch-dual-trust.yaml
talosctl patch machineconfig --nodes 192.168.1.12 --patch @patch-dual-trust.yaml
```

### Phase 3: Issue New Certificates Under New CA

This phase depends on which CA you are rotating. For Kubernetes CA, you must update the API server, kubelet, and controller-manager configurations to use the new CA for signing.

### Phase 4: Switch Primary CA

Update machine configs to use the new CA as the primary signing CA while keeping the old CA in the trust bundle temporarily.

### Phase 5: Remove Old CA from Trust Bundle

Once all certificates in use are signed by the new CA, remove the old CA from the trust bundle.

```yaml
# patch-remove-old-ca.yaml
machine:
  acceptedCAs: []  # Remove the old CA — only the new primary CA remains
```

```bash
talosctl patch machineconfig --nodes 192.168.1.10 --patch @patch-remove-old-ca.yaml
# Repeat for all nodes
```

## Monitoring Certificate Expiry

### Check Current Certificate Expiry

```bash
# Check Talos API certificates
talosctl get certificates --nodes 192.168.1.10

# Check Kubernetes API server certificate expiry
echo | openssl s_client -connect 192.168.1.10:6443 -servername kubernetes 2>/dev/null | \
  openssl x509 -noout -dates

# Check Talos API certificate expiry
echo | openssl s_client -connect 192.168.1.10:50000 2>/dev/null | \
  openssl x509 -noout -dates -subject

# Check etcd certificate expiry
talosctl -n 192.168.1.10 get etcdcerts
```

### Check Client Certificate Expiry

```bash
# Check talosconfig client cert expiry
cat ~/.talos/config | yq '.contexts[].crt' | base64 -d | \
  openssl x509 -noout -dates

# Check kubeconfig client cert expiry
kubectl config view --raw -o jsonpath='{.users[0].user.client-certificate-data}' | \
  base64 -d | openssl x509 -noout -dates
```

### Alerting

Set up monitoring to alert when certificates approach expiry. Example Prometheus rule:

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: certificate-expiry
  namespace: monitoring
spec:
  groups:
    - name: certificate-expiry
      rules:
        - alert: KubernetesAPIServerCertExpiringSoon
          expr: apiserver_client_certificate_expiration_seconds_count > 0 and histogram_quantile(0.01, rate(apiserver_client_certificate_expiration_seconds_bucket[5m])) < 7776000
          for: 10m
          labels:
            severity: warning
          annotations:
            summary: "Kubernetes API client certificate expiring within 90 days"
            description: "A client certificate used to authenticate to the API server will expire soon."

        - alert: KubernetesAPIServerCertExpiryCritical
          expr: apiserver_client_certificate_expiration_seconds_count > 0 and histogram_quantile(0.01, rate(apiserver_client_certificate_expiration_seconds_bucket[5m])) < 2592000
          for: 10m
          labels:
            severity: critical
          annotations:
            summary: "Kubernetes API client certificate expiring within 30 days"
            description: "A client certificate used to authenticate to the API server is about to expire. Rotate immediately."
```

### Expiry Check Script

```bash
#!/usr/bin/env bash
set -euo pipefail

# Check Kubernetes API server cert
echo "=== Kubernetes API Server Certificate ==="
echo | openssl s_client -connect 192.168.1.10:6443 -servername kubernetes 2>/dev/null | \
  openssl x509 -noout -dates -subject

echo ""
echo "=== Talos API Certificate ==="
echo | openssl s_client -connect 192.168.1.10:50000 2>/dev/null | \
  openssl x509 -noout -dates -subject

echo ""
echo "=== kubeconfig Client Certificate ==="
kubectl config view --raw -o jsonpath='{.users[0].user.client-certificate-data}' | \
  base64 -d | openssl x509 -noout -dates -subject 2>/dev/null || echo "No client cert in kubeconfig (may use token auth)"

echo ""
echo "=== talosconfig Client Certificate ==="
grep "crt:" ~/.talos/config | head -1 | awk '{print $2}' | base64 -d | \
  openssl x509 -noout -dates -subject 2>/dev/null || echo "Could not parse talosconfig cert"
```

## Key Reminders

- **10-year default lifetime** — certificates generated at cluster creation expire 10 years later. Set a calendar reminder.
- **`talosctl rotate-ca` is the safe path** — it handles the dual-trust transition automatically. Manual rotation is error-prone.
- **Always dry-run first** — `--dry-run` shows exactly what will change without touching the cluster.
- **Distribute new configs after rotation** — old `talosconfig` and `kubeconfig` files become invalid after rotation completes.
- **Back up `secrets.yaml` after rotation** — the new CA keys need to be in your backup set.
- **Talos does not support CRL/OCSP** — the only way to revoke a certificate is to rotate the entire CA.
